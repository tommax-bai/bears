## Context

这是一个从零开始的能力提案，当前仓库没有既有业务代码或约束可复用。目标产品包含两个明显不同的使用面：一个给访客使用的移动端留言板，一个给管理员使用的审核后台；两者共享同一套留言数据，但在权限、状态可见性和交互密度上差异很大。

留言板的核心风险不在页面展示，而在内容治理。任何允许公开提交文本的系统都需要提前定义审核流、限流与审计，否则很容易在上线后被垃圾内容或恶意刷屏击穿。设计上因此采用“移动优先前台 + 状态驱动后台 + 最小可行风控”的结构。

## Goals / Non-Goals

**Goals:**
- 提供单手可用、首屏信息清晰的手机留言体验。
- 让管理员能在手机和桌面上完成高频审核动作。
- 用明确的留言状态机隔离“提交”“待审”“发布”“拒绝”“屏蔽”等流程。
- 以最小实现覆盖基础反垃圾、审计留痕和可运营性。

**Non-Goals:**
- 不包含复杂社交功能，例如回复楼中楼、点赞、私信或用户关系链。
- 不包含富文本、图片上传或附件管理。
- 不包含复杂权限系统，默认仅有管理员后台角色。
- 不包含推荐算法或高级 BI 报表。

## Decisions

### 1. 采用移动优先的双端架构

前台留言板按窄屏优先设计，默认信息层级为“最新留言列表 + 固定底部发布入口 + 提交反馈”。后台则使用响应式两栏/单栏切换，在手机上将筛选与操作收纳为抽屉，在桌面上保留列表和详情并排。

选择这个方案而不是先做桌面优先页面再压缩到手机，是因为留言板的主要访问场景已经明确偏向手机，后续从手机扩展到桌面比反向裁剪更稳定。

### 2. 使用显式状态机管理留言生命周期

留言状态定义为 `pending`、`approved`、`rejected`、`hidden`。访客提交后默认进入 `pending`；只有 `approved` 的内容对外公开；`hidden` 用于已发布后下架；`rejected` 仅对后台可见。

选择显式状态，而不是只用一个“是否发布”布尔值，是因为后台还需要支持复审、统计和操作历史。布尔值无法表达被拒绝与被下架之间的业务差异。

### 3. 提前内建轻量风控，不把审核完全留给人工

提交侧应内建字数限制、必填校验、频率限制、敏感词命中拦截和重复提交保护；后台保留人工复核与封禁入口。风控只做基础规则，不引入重型第三方内容审核依赖。

这个选择控制了实现复杂度。对一个 MVP 留言板来说，先用规则化防护建立底线，比过早接入复杂审核服务更合适。

### 4. 后台操作必须可追踪

管理员的审核、拒绝、隐藏、恢复等动作都记录操作人、时间、备注和前后状态。列表页展示最近处理结果，详情页展示完整审计轨迹。

选择完整留痕，是为了降低误操作和多人协作下的定位成本，也为后续争议内容复盘提供依据。

## Risks / Trade-offs

- [匿名留言增加垃圾内容风险] -> 通过提交频控、敏感词规则、默认待审与后台批量处理降低风险。
- [移动端后台信息密度受限] -> 将高频动作前置，次级字段折叠到详情抽屉，避免手机审核负担过重。
- [纯规则风控存在误杀与漏判] -> 支持人工恢复、拒绝备注和规则迭代，不把自动拦截作为唯一决策。
- [从零实现缺少现有设计系统] -> 先定义清晰布局与状态约束，后续实现时再选择具体技术栈和组件库。

## Migration Plan

1. 先落地数据模型、状态机和后台审核 API，保证治理链路成立。
2. 再实现移动端留言板的浏览与提交流程，并仅暴露已审核内容。
3. 用种子数据验证手机提交流程、后台筛选与批量审核闭环。
4. 上线初期保守配置限流和敏感词规则，依据真实内容逐步调整。
5. 若出现高误判，可临时放宽自动拦截规则，但保留默认待审机制作为回退。

## Open Questions

- 留言板是否允许完全匿名，还是至少要求昵称加验证码。
- 管理后台是否需要区分“超级管理员”和“审核员”两级权限。
- 是否需要在前台展示精选置顶留言，而不仅仅是按时间倒序。
- 是否需要接入短信、邮件或站内提醒，通知管理员有新留言待审。
